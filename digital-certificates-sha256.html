What Are Digital Certificates? Cryptography Lab

A digital certificate is an electronic document that binds a public key to an identity — a person, organisation, or domain. Certificates follow the X.509 standard and are issued and signed by a Certificate Authority (CA).

When your browser connects to a website over HTTPS, it validates the server's certificate: verifying the CA's signature, checking the validity period, and confirming the certificate hasn't been revoked. This chain of trust underpins nearly all secure communication on the internet.

// certificate anatomy

Reading an X.509 Certificate

An X.509 certificate contains a subject (who it belongs to), an issuer (who signed it), a public key, a validity window, and a digital signature from the CA. OpenSSL's x509 command lets you inspect every field.

# View a certificate in human-readable form openssl x509 -in certificate.pem -text -noout # Key fields in the output Subject: CN=example.com, O=Example Corp, C=US Issuer: CN=DigiCert Global CA, O=DigiCert Inc Validity: Not Before: Jan 1 00:00:00 2024 GMT Not After: Jan 1 00:00:00 2025 GMT Public Key Algorithm: rsaEncryption (2048 bit) Signature Algorithm: sha256WithRSAEncryption
FieldPurpose
SubjectWho the certificate belongs to (domain, org, country)
IssuerThe CA that signed and vouches for this certificate
ValidityNot Before / Not After — the certificate's active window
Public KeyThe public half of the key pair for this identity
SignatureCA's cryptographic signature over all of the above
// sha-256

The Hash Function Underneath

SHA-256 takes any input and produces a fixed 256-bit (32-byte) output. Two properties make it cryptographically useful: it is computationally infeasible to reverse, and even a one-character change in input produces a completely different output — the avalanche effect.

Certificates rely on SHA-256 for the CA's signature: the CA hashes the certificate contents, then encrypts that hash with its private key. Anyone with the CA's public key can decrypt and verify the hash matches — confirming nothing has been tampered with.

# Hash a string with SHA-256 echo -n "Jordan Flaum" | openssl dgst -sha256 # Hash a file openssl dgst -sha256 certificate.pem # Even one character changes everything (avalanche effect) echo -n "Jordan Flaum" | openssl dgst -sha256 # → a3f9c2... echo -n "jordan Flaum" | openssl dgst -sha256 # → 8b17e4... (completely different)
Avalanche Effect: SHA-256("Jordan Flaum") → a deterministic fingerprint. Change a single character and the output is unrecognisable.

PKI Hierarchy

Real-world PKI uses a hierarchy: a root CA signs intermediate CAs, which sign end-entity certificates. Browsers and operating systems ship with a built-in trust store of root CAs. If you can trace a certificate's signature chain up to a trusted root, the certificate is considered valid.

# Verify a full certificate chain openssl verify -CAfile root-ca.pem -untrusted intermediate.pem server.pem # Inspect the chain from a live TLS connection openssl s_client -connect example.com:443 -showcerts
